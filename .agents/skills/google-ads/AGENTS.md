# Global Rules

## Saarthi Platform Execution Constraint (READ-ONLY AI)
- **Strict Prohibition on Direct Mutations:** NO agent is allowed to directly execute any edit, update, mutate, pause, or enable operation on the Google Ads portal or any dashboard. The AI is strictly limited to fetching data and generating recommendations.
- **Human Execution Only:** The Saarthi platform is designed to assist humans. All execution operations (budget changes, bid adjustments, keyword additions/pauses, ad launches, campaign status changes) MUST be surfaced as actionable recommendations or prepared API payloads within the Saarthi UI.
- **Execution Workflow:** Agents prepare the data and the suggested changes. A human operator reviews these suggestions and manually clicks "Execute" to push the changes. The AI never bypasses this human-in-the-loop mechanism, given the financial stakes (Millions of Rupees).
- **No auto-apply of Google's own recommendations:** `RecommendationService.ApplyRecommendation` (and any other mutate/apply RPC) is a direct-mutation call. No agent may invoke it. Google's `recommendation` resource may only be *read* and surfaced as a suggestion, same as any agent-generated one.

## Anti-Hallucination & Data Integrity Rules (Global — applies to every agent in this directory)
These exist because a wrong number stated with confidence is more dangerous than "I don't know" — the account owner acts on what Saarthi tells them, with real budget on the line.

1. **Never invent a Google Ads API resource or field name.** Only use fields explicitly listed in the invoking agent's own `SKILL.md`. If a query needs a field that isn't listed there, verify it first via `GoogleAdsFieldService.GetGoogleAdsField` / `SELECT ... FROM google_ads_field WHERE ...` (see `universal/gaql-query-agent`) rather than guessing a plausible-sounding name. Field availability also changes across API versions — do not assume a field from a past version is still valid.
2. **Never fabricate a metric, count, or trend to fill a gap.** If a GAQL query returns zero rows, an error, or a partial failure, report that fact explicitly ("no rows returned for campaign X in range Y") — do not estimate, interpolate, or "round to a plausible number."
3. **Every quantitative claim must cite its source**: which GAQL query, which resource, which date range. An output that can't be traced back to a specific query result must be labeled "insufficient data" instead of answered from general knowledge about how Google Ads "usually" behaves.
4. **GAQL does not support arbitrary joins.** Only resource/attribute/segment/metric combinations documented as compatible for a given `FROM` resource are queryable together (check via `google_ads_field.selectable_with` / `google_ads_field.segments`). Do not assume two resources can be combined in one query just because both contain relevant data — two separate queries may be required.
5. **Frame Google's own `recommendation` impact estimates as unverified projections, not guarantees** — Google itself does not warrant these numbers; agents must not upgrade that framing to a promise when reporting to the user.
6. **State campaign-type visibility limits explicitly rather than substituting a workaround.** Example: Performance Max does not expose search-term-level data the way standard Search campaigns do (limited instead to `campaign_search_term_insight` category-level insights) — an agent must say "not available for PMax" rather than quietly reporting Search-campaign-style search terms as if they applied.
7. **Do not claim causality from correlation.** If asked "did performance improve after change X," report what changed and what the metrics did in the same window, but do not assert the change *caused* the movement unless corroborated by an isolating mechanism (e.g. an experiment/A-B test arm from `campaign/experiment-ab-testing-agent`) — see `universal/memory-learning-agent`.

## API Version & Change Management (Global)
Google Ads API versions are semantically versioned; major releases can remove or rename fields, and Google publishes a deprecation schedule. Every agent in this directory is vulnerable to the same failure mode: a field or enum that was valid when its `SKILL.md` was written silently stops being valid after an API upgrade.

1. **Treat every field name in a `SKILL.md` as a snapshot, not a permanent guarantee.** When a query against a field that previously worked starts failing or returning unexpected nulls, the first hypothesis should be "this field changed or was deprecated," not "the account has no data" — re-verify via `google_ads_field` (`universal/gaql-query-agent`) before concluding either way.
2. **Prefer resource/field metadata lookups over hardcoded assumptions** wherever a `SKILL.md` already flags a field as "verify before use" (Local, Demand Gen, and PMax search-term-insight fields are the current examples in this directory) — those flags exist because that part of the schema has changed before and is more likely to change again.
3. **Do not hardcode a specific API version number in a query or payload.** Use the client library's current default/constants rather than a literal `vNN` string, so a version bump doesn't require hunting through every agent's logic.
4. **When an agent's expected field set and the live API's actual field set diverge, report the divergence explicitly** (e.g. "expected field X is not present in this API version's schema for resource Y") rather than silently working around it or guessing a replacement field — the human maintaining this library needs that signal to update the affected `SKILL.md`.
